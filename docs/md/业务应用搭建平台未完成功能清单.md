# 业务应用搭建平台未完成功能清单

> 生成日期：2026-06-17  
> 当前状态判断基于本分支已有代码：`youlai-flowable` 后端模块、`vue-admin/src/views/app-builder`、`vue-admin/src/views/workflow`。

## 1. 当前已完成到什么程度

当前分支已经补出了一版“业务应用搭建平台基础壳”，主要包括：

| 模块 | 当前状态 |
| --- | --- |
| 应用管理 | 有应用分页、创建、编辑、发布、停用、复制、删除基础接口和页面 |
| 数据建模 | 有模型、字段配置基础接口和页面 |
| 表单设计器 | 有拖拽设计、预览、本地导出、后端保存、发布、按 `formKey` 加载 |
| 页面配置 | 有列表/详情/表单页面配置的基础 JSON Schema 保存与发布 |
| 页面运行 | 已新增发布页面运行入口 `/app-builder/runtime/:pageId`，可按页面 Schema 渲染查询、表格和增删改 |
| 业务数据 | 有基于 JSON 存储的新增、编辑、删除、查询、Excel/JSON 导入导出，并同步镜像到动态物理业务表 |
| 工作流 | 有 Flowable 模型、流程设计、发布、发起、待办、已办、详情、审批记录 |
| 表单与流程打通 | 流程可绑定 `formKey`，业务数据可提交审批，流程完成/驳回/撤回/终止会反写业务数据状态 |
| 扩展中心 | 有报表运行预览、API 参数映射/鉴权/日志、自动化数据触发执行、模板完整复制、版本快照回滚、多租户、日志页面 |

但这些能力多数还处在“基础 CRUD + 配置存储”阶段，距离 PRD 中完整平台仍有明显差距。

## 2. P0/MVP 仍未完成或未完全闭环

### 2.1 动态建表能力未做

当前业务数据采用 `app_builder_data.data_json` 存储，尚未根据数据模型自动创建或变更物理业务表。

未完成项：

| 功能 | 当前状态 | 待补内容 |
| --- | --- | --- |
| 根据模型建表 | 已补 | 模型发布和字段保存时生成物理表 |
| 字段增删改同步表结构 | 部分完成 | 已支持新增字段执行 `ALTER TABLE ADD COLUMN`，字段删除/改名暂不自动 DROP，避免历史数据损坏 |
| 字段类型映射 | 已补 | 已建立常用字段到 MySQL 类型映射 |
| 字段版本管理 | 未做 | 保留历史字段版本，避免历史数据损坏 |
| 动态索引 | 部分完成 | 已支持主字段索引 |
| 模型删除安全检查 | 已补 | 有业务数据、表单、页面、报表、自动化、流程绑定时禁止硬删；无依赖时同步删除字段配置 |

建议优先级：高。

### 2.2 页面设计器还只是配置表单，不是真正可视化设计器

当前页面配置只保存 `page_schema` JSON，没有真正的可视化列表页/详情页/编辑页设计器。

未完成项：

| 功能 | 当前状态 | 待补内容 |
| --- | --- | --- |
| 列表页可视化配置 | 部分做了 JSON 配置 | 查询条件、表格列、按钮、批量操作可视化 |
| 详情页配置 | 仅有 JSON 保存 | 分组、布局、字段显隐、只读规则 |
| 编辑页配置 | 仅有 JSON 保存 | 与表单设计器联动 |
| 页面预览 | 已补 | 页面配置列表和编辑弹窗支持按 `page_schema` 实时预览查询区、表格列和按钮 |
| 页面发布后运行入口 | 已补 | 已新增 `/app-builder/runtime/:pageId` 运行页 |
| 菜单自动挂载 | 已补 | 页面发布、应用发布、应用停用、页面删除时同步 `app_builder_menu` 运行菜单/动态路由清单 |

建议优先级：高。

### 2.3 权限中心未真正接入应用搭建平台

系统已有用户、角色、部门、菜单、按钮、数据权限基础，但应用搭建平台自己的权限模型还没有完整落地。

未完成项：

| 权限类型 | 当前状态 | 待补内容 |
| --- | --- | --- |
| 应用权限 | 未做 | 控制用户可访问哪些应用 |
| 页面权限 | 未做 | 控制用户可访问哪些业务页面 |
| 按钮权限 | 系统已有基础，但未绑定业务页面 | 新增、编辑、删除、导入、导出按钮权限 |
| 字段权限 | 未做 | 字段可见、可编辑、脱敏 |
| 数据权限 | 系统已有部门/角色基础，但未用于业务数据 | 本人、本部门、部门及下级、自定义范围 |

建议优先级：高。

### 2.4 应用发布闭环不完整

### 2.5 业务数据审批状态未与流程结果联动

当前流程可以保存发起表单数据，但业务数据和流程实例之间没有完整状态同步。

未完成项：

| 功能 | 当前状态 | 待补内容 |
| --- | --- | --- |

## 6. 建议后续实施顺序

### 第一优先级：把 MVP 闭环做实

1. 权限中心接入应用、页面、按钮、字段、数据范围。
2. 页面配置变成真实运行页面，而不是只保存 JSON。
3. 动态建表的字段版本、删除/变更策略和索引策略。
4. 报表、API、自动化的生产级安全和重试机制。
5. 租户隔离、测试与部署脚本。

### 第二优先级：把 P1 能力变成可用

1. 报表运行页和 ECharts 渲染。
2. API 参数映射、鉴权、Webhook、调用日志增强。
3. 自动化执行器，接入数据事件、流程事件、定时任务。
4. 操作日志自动埋点。

### 第三优先级：增强体验和商业化能力

1. 模板复制完整应用。
2. AI 真正接入大模型。
3. 版本对比和真实回滚。
4. 移动端 H5。
5. 多租户真实隔离和套餐限制。

---

## 7. 总结

当前分支已经完成了“功能入口和基础数据结构”的大部分铺设，但距离一个真正可交付的业务应用搭建平台，还差以下关键能力：

1. **权限和安全闭环**：应用、页面、按钮、字段、数据范围权限还没有真正落到业务运行态。
2. **数据模型的工程化**：动态建表已补基础能力，但字段版本、删除/变更策略、索引和查询性能还要加强。
3. **扩展能力生产化**：API 鉴权、Webhook 回调、重试、定时任务、消息通知还需要继续补。
4. **体验与商业化能力**：模板内置示例、移动端 H5、多租户套餐、AI 真模型接入还没完成。
5. **安全、权限、测试、部署**：离生产可用还需要系统性补强。
