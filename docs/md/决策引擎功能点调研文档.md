# 决策引擎功能点调研文档

## 1. 调研目的

本文用于梳理市面上主流决策引擎、规则引擎、风控决策平台、DMN 决策建模平台的常见功能点，为后续产品设计、PRD 编写、系统架构设计提供参考。

决策引擎的核心价值是：将业务判断逻辑从应用代码中抽离出来，通过可配置、可测试、可发布、可追溯的方式，实现自动化决策。

常见应用场景包括：

* 风控反欺诈：交易拦截、注册风险识别、登录风险识别、设备风险识别
* 信贷审批：授信额度、准入规则、评分卡、反欺诈规则
* 营销决策：用户分群、优惠券发放、权益匹配、活动准入
* 业务审批：自动通过、自动拒绝、转人工审核
* 运营策略：价格策略、库存策略、客户分层、服务优先级
* 流程自动化：在 BPM、低代码、工作流系统中作为条件判断和策略执行组件

## 3. 决策引擎核心功能模块

### 3.1 规则管理模块

规则管理是决策引擎的基础模块，用于创建、维护和执行业务规则。

#### 功能点

| 功能点     | 说明                         |
| ------- | -------------------------- |
| 规则新增    | 支持创建单条业务规则                 |
| 规则编辑    | 支持修改规则名称、条件、动作、优先级、生效状态    |
| 规则复制    | 支持复制已有规则快速创建新规则            |
| 规则删除    | 支持逻辑删除或停用规则                |
| 规则启停    | 支持规则启用、停用、灰度启用             |
| 规则分类    | 按业务场景、风险类型、产品线、渠道分类        |
| 规则标签    | 给规则打标签，如“高风险”“营销”“审批”“反欺诈” |
| 规则优先级   | 多条规则同时命中时按优先级执行            |
| 规则冲突检测  | 检查规则之间是否存在互斥、重复、覆盖问题       |
| 规则完整性校验 | 检查规则条件是否缺失、动作是否为空、变量是否不存在  |
| 规则依赖分析  | 查看规则使用了哪些变量、模型、数据源         |
| 规则搜索    | 按规则名称、编码、标签、创建人、状态搜索       |

#### 规则表达方式

* IF / THEN 规则
* 决策表
* 决策树
* 评分卡
* 规则集
* 规则流
* DMN 决策模型
* 表达式规则
* 脚本规则
* SQL 规则
* 自定义函数规则

### 3.2 规则条件配置模块

规则条件用于描述“什么情况下触发规则”。

#### 功能点

| 功能点    | 说明                                           |
| ------ | -------------------------------------------- |
| 条件字段选择 | 从变量库中选择字段，如年龄、金额、城市、设备 ID                    |
| 操作符配置  | 支持 =、!=、>、<、>=、<=、in、not in、contains、between |
| 多条件组合  | 支持 AND、OR、嵌套条件组                              |
| 空值判断   | 支持 is null、is not null                       |
| 时间条件   | 支持近 N 天、当天、指定时间范围                            |
| 次数统计   | 如 1 小时内登录失败次数 > 5                            |
| 金额统计   | 如 24 小时内交易金额累计 > 10000                       |
| 黑白名单判断 | 判断手机号、身份证、IP、设备是否命中名单                        |
| 正则匹配   | 支持手机号、邮箱、地址等格式校验                             |
| 自定义表达式 | 支持通过表达式语言编写复杂条件                              |
| 复杂对象取值 | 支持 JSON、Map、List、对象嵌套字段取值                    |

### 3.3 规则动作模块

规则动作定义命中规则后系统应该做什么。

#### 功能点

| 动作类型   | 示例                     |
| ------ | ---------------------- |
| 返回决策结果 | 通过、拒绝、转人工、待补充材料        |
| 设置风险等级 | 低风险、中风险、高风险            |
| 设置评分   | 加分、扣分、计算总分             |
| 命中标签   | 标记“疑似羊毛党”“高价值客户”“异常设备” |
| 触发通知   | 发送短信、邮件、站内信、Webhook    |
| 调用接口   | 调用外部风控、征信、营销、审批系统      |
| 写入名单   | 加入黑名单、白名单、观察名单         |
| 进入人工审核 | 创建审核任务并分配给审核人员         |
| 阻断业务流程 | 拒绝交易、禁止登录、禁止提现         |
| 进入后续策略 | 跳转到下一个规则集或决策流节点        |

### 3.4 规则集管理模块

规则集用于将多条规则组织成一个可执行单元。

#### 功能点

| 功能点   | 说明                |
| ----- | ----------------- |
| 规则集创建 | 创建一个业务场景下的规则集合    |
| 规则集编排 | 配置规则执行顺序          |
| 串行执行  | 前一条规则执行后再执行下一条    |
| 并行执行  | 多条规则同时判断，汇总结果     |
| 短路执行  | 命中拒绝规则后不再继续执行     |
| 命中全部  | 所有规则满足才触发结果       |
| 命中任一  | 任意一条规则满足即可触发结果    |
| 权重汇总  | 不同规则按权重累加风险分      |
| 优先级决策 | 多个结果按优先级输出最终结果    |
| 规则集版本 | 支持规则集版本管理         |
| 规则集发布 | 将规则集发布到测试、预发、生产环境 |

### 3.5 决策流编排模块

决策流是决策引擎的核心能力，用于把规则、模型、数据、动作串联成完整决策链路。

#### 功能点

| 功能点     | 说明                 |
| ------- | ------------------ |
| 可视化流程编排 | 拖拽方式配置决策节点         |
| 开始节点    | 定义决策入口             |
| 规则节点    | 执行规则或规则集           |
| 条件分支节点  | 根据条件进入不同分支         |
| 模型节点    | 调用评分模型、机器学习模型      |
| 数据查询节点  | 查询数据库、接口、缓存、画像标签   |
| 计算节点    | 执行变量计算、公式计算        |
| 人工审核节点  | 转入人工审核流程           |
| 动作节点    | 执行通知、拦截、写库、接口调用    |
| 结束节点    | 输出最终决策结果           |
| 节点拖拽排序  | 支持调整节点执行顺序         |
| 节点复制    | 快速复用节点配置           |
| 节点启停    | 支持单个节点启用/停用        |
| 分支合并    | 多条分支汇总为最终结果        |
| 异常分支    | 某节点异常时进入兜底策略       |
| 超时控制    | 外部接口或模型调用超时后执行降级策略 |

### 3.6 变量管理模块

变量是规则判断和决策计算的基础。

#### 功能点

| 功能点    | 说明                           |
| ------ | ---------------------------- |
| 变量定义   | 创建业务变量，如 userAge、orderAmount |
| 变量分类   | 基础变量、衍生变量、统计变量、外部变量、模型变量     |
| 变量类型   | 字符串、数字、布尔、日期、枚举、对象、数组        |
| 变量来源   | 请求入参、数据库、接口、缓存、标签系统、模型输出     |
| 变量默认值  | 当变量为空时使用默认值                  |
| 变量计算公式 | 如近 7 日交易金额 / 近 7 日交易次数       |
| 变量依赖关系 | 查看变量依赖了哪些字段或接口               |
| 变量版本管理 | 变量口径变更后保留历史版本                |
| 变量权限控制 | 不同角色可查看/使用不同变量               |
| 变量使用分析 | 查看变量被哪些规则、模型、决策流引用           |
| 变量测试   | 输入样例数据验证变量计算结果               |

### 3.7 数据源管理模块

决策引擎通常需要接入多种数据源，为规则判断提供事实数据。

#### 功能点

| 数据源类型              | 说明                      |
| ------------------ | ----------------------- |
| 请求入参               | 业务系统调用决策引擎时传入的数据        |
| MySQL / PostgreSQL | 查询业务库、用户库、订单库           |
| Redis              | 查询实时缓存、计数器、黑白名单         |
| Elasticsearch      | 查询日志、行为轨迹、历史记录          |
| 第三方 API            | 征信、设备指纹、实名认证、IP 风险      |
| 文件数据               | CSV、Excel、离线名单          |
| 标签系统               | 用户标签、设备标签、画像标签          |
| 模型服务               | 调用机器学习模型返回评分            |
| 消息队列               | 接收实时事件，如 Kafka、RocketMQ |

#### 数据源功能点

* 数据源新增、编辑、删除
* 连接测试
* 接口鉴权配置
* 超时时间配置
* 重试次数配置
* 失败降级策略
* 数据字段映射
* 数据缓存配置
* 数据脱敏配置
* 数据访问日志

#### 功能点

| 功能点     | 说明                     |
| ------- | ---------------------- |
| 表格化规则配置 | 用类似 Excel 的方式维护规则      |
| 条件列     | 配置输入条件                 |
| 结果列     | 配置输出结果                 |
| 优先级列    | 决定多行命中时的执行顺序           |
| 命中策略    | 支持唯一命中、首个命中、全部命中、优先级命中 |
| 批量导入    | 支持 Excel 导入规则          |
| 批量导出    | 支持导出决策表                |
| 单元格校验   | 校验格式、枚举值、范围            |
| 重叠检测    | 检查多行条件是否重复或冲突          |
| 缺口检测    | 检查是否存在未覆盖的条件范围         |
| 版本对比    | 比较两个版本的决策表差异           |

#### 功能点

| 等级映射  | 分数映射为 A/B/C/D 等级  |
| 决策映射  | 分数映射为通过、拒绝、人工审核   |

| 评分卡发布 | 发布到生产环境执行         |

#### 功能点

| 功能点    | 说明              |
| ------ | --------------- |
| 模型注册   | 注册外部模型服务        |
| 模型调用   | 在决策流中调用模型       |
| 模型输入映射 | 将变量映射为模型入参      |

| 模型兜底   | 模型失败时走默认规则      |
| 模型解释   | 展示模型评分原因、特征贡献   |

| 路径追踪   | 查看决策流执行路径           |
| 输出解释   | 展示最终结果由哪些规则/模型决定    |
| 测试报告   | 生成命中率、拒绝率、通过率、人工率报告 |

## 10. 产品设计重点

做决策引擎时，最关键的不是“能不能写规则”，而是以下几点：

1. 业务人员能不能看懂规则
2. 规则变更是否不需要发版
3. 规则上线前能不能测试和仿真
4. 规则上线后能不能监控效果
5. 出问题后能不能追溯原因
6. 多条规则冲突时能不能稳定输出结果
7. 外部数据源异常时是否有兜底策略
8. 模型和规则能不能组合使用
9. 是否支持版本、审批、回滚
10. 是否能解释每一次决策结果

## 11. 总结

市面上成熟的决策引擎通常不是单纯的 if-else 配置工具，而是一套完整的“策略生产、测试、发布、执行、监控、审计”平台。

如果要建设一个通用型决策引擎，建议优先完成：

* 规则管理
* 变量管理
* 规则集管理
* 决策流编排
* 实时决策 API
* 测试仿真
* 版本发布
* 执行日志
* 监控审计

如果是风控场景，还需要重点增强：

* 实时事件管理
* 黑白名单
* 设备/IP/用户画像变量
* 评分卡
* 模型融合
* 历史回放
* 策略仿真
* 命中解释
* 人工审核流转

最终目标是让业务策略可以低代码配置、实时执行、可解释、可回滚、可持续优化。
