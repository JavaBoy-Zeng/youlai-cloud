# MQ 消息可靠性、重复消费、顺序消息面试文档

## 一、面试核心回答

MQ 相关问题主要围绕三个核心点展开：

```text
1. 消息可靠性：如何保证消息不丢？
2. 重复消费：如何保证业务不会被重复执行？
3. 顺序消息：如何保证同一业务维度的消息有序？
```

可以用一句话总结：

> MQ 为了保证消息不丢，通常会采用至少一次投递机制，因此消费者必须做好幂等处理；如果业务有顺序要求，一般通过相同业务 key 进入同一个队列，并使用顺序消费来保证局部有序。

---

# 二、消息可靠性：如何保证消息不丢？

## 1. 消息发送链路

一条 MQ 消息完整链路如下：

```text
生产者 -> MQ Broker -> 消费者 -> 业务处理完成
```

所以消息可靠性不能只看 MQ 本身，要从以下几个阶段分析：

```text
1. 生产者发送可靠
2. 业务数据和消息一致性
3. Broker 存储可靠
4. 消费者消费可靠
```

---

## 2. 生产者发送可靠

生产者发送消息时，可能出现网络异常、Broker 不可用、发送超时等问题。

如果生产者只是把消息发出去，不关心结果，这种方式叫做：

```text
fire-and-forget
```

意思是：

```text
发出去就不管了
```

这种方式性能可能比较高，但是可靠性差。

例如：

```java
producer.send(message);
// 不判断发送结果
// 不重试
// 不记录失败日志
```

如果发送失败，业务系统可能完全不知道，导致消息丢失。

正确做法是：

```text
同步发送 + 判断发送结果 + 失败重试 + 记录日志/补偿
```

以 RocketMQ 为例：

```java
SendResult result = producer.send(message);

if (result.getSendStatus() != SendStatus.SEND_OK) {
    // 记录日志
    // 重试
    // 或者写入本地消息表，后续补偿发送
}
```

面试表达：

> 生产端不能简单 fire-and-forget，而是要关注 MQ 返回的发送结果。如果发送失败，可以进行重试，或者将消息落入本地消息表，由定时任务后续补偿发送。

---

## 3. 业务数据和 MQ 消息一致性

一个常见问题是：

```text
业务数据保存成功了，但是 MQ 消息发送失败了
```

例如：

```text
订单创建成功 -> 扣库存消息发送失败
```

这样会导致订单系统和库存系统数据不一致。

常见解决方案有两种：

```text
1. 本地消息表
2. 事务消息
```

---

## 4. 本地消息表方案

本地消息表是比较常见、稳定、容易落地的方案。

核心思想：

```text
业务数据和消息记录放在同一个本地事务中提交
```

流程如下：

```text
开启事务
  保存业务数据，比如创建订单
  保存消息记录，状态为 INIT
提交事务

定时任务扫描 INIT 消息
  发送 MQ
  发送成功后更新消息状态为 SUCCESS
  发送失败则继续重试
```

伪代码：

```java
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);

    MessageLog messageLog = new MessageLog();
    messageLog.setBizId(order.getId());
    messageLog.setTopic("stock-topic");
    messageLog.setBody("扣减库存消息");
    messageLog.setStatus("INIT");

    messageLogMapper.insert(messageLog);
}
```

定时补偿：

```java
List<MessageLog> list = messageLogMapper.selectInitMessages();

for (MessageLog message : list) {
    try {
        producer.send(buildMessage(message));
        messageLogMapper.updateStatus(message.getId(), "SUCCESS");
    } catch (Exception e) {
        // 记录失败次数，下次继续补偿
    }
}
```

面试表达：

> 本地消息表主要解决本地事务和 MQ 消息发送之间的一致性问题。业务数据和消息表在同一个数据库事务中提交，只要事务提交成功，消息就不会丢。即使 MQ 临时发送失败，也可以通过定时任务扫描消息表进行补偿发送。

---

## 5. RocketMQ 事务消息

RocketMQ 支持事务消息，适合解决本地事务和消息发送之间的最终一致性问题。

流程如下：

```text
1. 生产者发送半消息
2. Broker 暂存半消息，此时消费者不可见
3. 生产者执行本地事务
4. 本地事务成功，提交消息
5. 本地事务失败，回滚消息
6. 如果生产者没有返回结果，Broker 会回查本地事务状态
```

面试表达：

> RocketMQ 事务消息解决的是本地事务和消息发送的最终一致性问题。它不是传统意义上的分布式事务，而是通过半消息、本地事务、事务回查机制，保证本地事务成功后消息最终可以被投递。

---

# 三、Broker 存储可靠性

Broker 收到消息后，也可能因为宕机、磁盘故障、主节点故障导致消息丢失。

Broker 端可靠性主要看两个维度：

```text
1. 刷盘策略
2. 主从复制策略
```

---

## 1. 刷盘策略

刷盘指的是消息是否真正写入磁盘。

RocketMQ 常见刷盘策略：

```text
SYNC_FLUSH   同步刷盘
ASYNC_FLUSH  异步刷盘
```

### 同步刷盘

```text
消息写入磁盘后，才返回发送成功
```

优点：

```text
可靠性更高
```

缺点：

```text
性能较低，响应时间更长
```

### 异步刷盘

```text
消息先写入内存，然后异步刷入磁盘
```

优点：

```text
性能更高，吞吐量更好
```

缺点：

```text
极端情况下，Broker 宕机可能丢失少量还没来得及刷盘的消息
```

---

## 2. 主从复制策略

主从复制指的是消息写入主节点后，是否同步到从节点。

RocketMQ 常见主从复制策略：

```text
SYNC_MASTER   主从同步复制
ASYNC_MASTER  主从异步复制
```

### 主从同步复制

```text
消息写入主节点后，还要同步到从节点，才返回成功
```

优点：

```text
可靠性更高
```

缺点：

```text
性能较低
```

### 主从异步复制

```text
消息写入主节点后立即返回成功，之后异步复制到从节点
```

优点：

```text
性能更好
```

缺点：

```text
如果主节点宕机，而消息还没来得及同步到从节点，可能会丢失少量消息
```

---

## 3. 最可靠组合

如果业务对消息可靠性要求很高，可以使用：

```text
同步刷盘 + 主从同步复制
```

也就是：

```text
SYNC_FLUSH + SYNC_MASTER
```

面试表达：

> Broker 端可靠性主要通过消息持久化和主从复制来保证。刷盘策略解决的是消息是否真正落盘的问题，主从复制解决的是主节点宕机后从节点是否还有这条消息的问题。如果要求可靠性最高，可以使用同步刷盘加主从同步复制，但性能会有所下降，需要结合业务场景取舍。

一句话记忆：

```text
同步刷盘：防止 Broker 宕机导致内存消息丢失
主从同步复制：防止主节点宕机后从节点没有这条消息
```

---

# 四、消费者消费可靠性

消费者拿到消息后，也可能因为业务异常、数据库异常、服务宕机导致消费失败。

正确消费流程：

```text
1. 消费者拉取/接收消息
2. 执行业务逻辑
3. 业务执行成功后 ACK
4. 业务执行失败则不 ACK 或返回失败
5. MQ 进行重试
6. 多次失败后进入死信队列
```

错误做法：

```text
刚收到消息就 ACK，然后再执行业务逻辑
```

如果 ACK 后业务失败，MQ 会认为消息已经消费成功，不会再投递，消息就丢了。

面试表达：

> 消费端要在业务逻辑执行成功之后再确认消息。如果消费失败，就返回失败或者不 ACK，让 MQ 进行重试。超过最大重试次数后，可以进入死信队列，再通过人工排查或者补偿任务处理。

---

# 五、重复消费：如何解决？

## 1. 为什么 MQ 会重复消费？

MQ 一般不能完全保证只消费一次，更多是保证至少投递一次。

也就是说：

```text
为了保证消息不丢，MQ 可能会把同一条消息投递多次
```

常见重复消费原因：

```text
1. 消费者业务处理成功了，但是 ACK 失败
2. 消费者消费超时，MQ 认为消费失败
3. 消费者消费过程中宕机，消息重新投递
4. 生产者发送超时后重试，导致同一业务消息发送多次
```

例如：

```text
库存已经扣减成功
但是 ACK 没有成功返回给 MQ
MQ 以为消费失败
于是又投递了一次
```

如果没有幂等控制，就会重复扣库存。

---

## 2. 重复消费的核心解决方案：幂等

面试一定要说：

> MQ 层面很难完全避免重复投递，所以业务侧必须做好幂等处理。

常见幂等方案：

```text
1. 数据库唯一索引
2. 消费记录表
3. Redis SETNX
4. 业务状态机
5. 乐观锁
```

---

## 3. 数据库唯一索引

这是最常见、最稳的方式。

可以建立消费记录表：

```sql
CREATE TABLE message_consume_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id VARCHAR(64) NOT NULL,
    consumer_group VARCHAR(64) NOT NULL,
    create_time DATETIME NOT NULL,
    UNIQUE KEY uk_msg_consumer (message_id, consumer_group)
);
```

消费时先插入消费记录：

```java
try {
    consumeLogMapper.insert(messageId, consumerGroup);
} catch (DuplicateKeyException e) {
    // 说明已经消费过，直接返回成功
    return;
}

doBusiness();
```

面试表达：

> 消费者可以先根据 messageId 或业务唯一 key 插入消费记录表，并建立唯一索引。如果插入成功，说明第一次消费，可以继续执行业务；如果插入失败，说明消息已经处理过，直接返回成功即可。

---

## 4. Redis SETNX

高并发场景下，也可以使用 Redis 的 SETNX 做幂等控制。

示例：

```java
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("mq:consume:" + messageId, "1", 10, TimeUnit.MINUTES);

if (!success) {
    return;
}

doBusiness();
```

但是 Redis SETNX 要注意一个问题：

```text
如果 key 设置成功了，但是业务执行失败了，后续重试可能会被误判为重复消费
```

所以更稳的做法是：

```text
Redis + 状态标记
或者数据库唯一索引
或者业务状态机
```

面试表达：

> Redis SETNX 可以用来做幂等，但要注意 key 设置成功后业务失败的问题。如果是核心业务，我更倾向于用数据库唯一索引或者业务状态机来保证幂等。

---

## 5. 业务状态机

对于订单、支付、发货这类业务，状态机是非常常见的幂等方案。

比如订单状态：

```text
待支付 -> 已支付 -> 已发货 -> 已完成
```

如果重复收到支付成功消息，可以这样处理：

```sql
UPDATE t_order
SET status = 'PAID'
WHERE order_id = ?
AND status = 'WAIT_PAY';
```

如果影响行数为 1，说明更新成功。

如果影响行数为 0，说明：

```text
1. 订单已经支付过
2. 当前状态不允许流转
3. 消息重复或者乱序
```

面试表达：

> 对于订单类业务，可以通过状态机保证幂等。比如只有待支付状态才能更新为已支付，重复收到支付成功消息时，由于状态已经是已支付，更新影响行数为 0，直接忽略即可。

---

# 六、顺序消息：如何保证顺序？

## 1. 顺序消息的核心思想

MQ 一般不保证全局顺序，只保证局部顺序。

面试要重点说明：

```text
实际项目中一般不追求全局顺序，而是保证同一个业务维度的局部顺序
```

例如：

```text
同一个订单的消息有序
同一个用户的消息有序
同一个账户的消息有序
同一辆车的轨迹消息有序
```

---

## 2. 为什么不推荐全局顺序？

全局顺序指所有消息都严格按照发送顺序消费。

实现方式：

```text
一个 Topic 只有一个 Queue
一个消费者线程串行消费
```

缺点：

```text
吞吐量低
并发能力差
性能瓶颈明显
```

所以实际项目中一般使用局部顺序。

---

## 3. 局部顺序如何实现？

以订单消息为例：

```text
订单创建
订单支付
订单发货
订单完成
```

要保证同一个订单的消息有序，需要保证：

```text
相同 orderId 的消息进入同一个 Queue
同一个 Queue 内由消费者串行消费
```

RocketMQ 示例：

```java
producer.send(message, (mqs, msg, arg) -> {
    Long orderId = (Long) arg;
    int index = (int) (orderId % mqs.size());
    return mqs.get(index);
}, orderId);
```

这样可以保证：

```text
orderId = 1001 的所有消息都进入同一个 Queue
orderId = 1002 的所有消息也进入固定的 Queue
```

消费者端使用顺序消费：

```java
MessageListenerOrderly
```

面试表达：

> 顺序消息一般通过业务 key 做队列选择，比如按照 orderId 取模，让同一个订单的消息始终进入同一个 Queue。然后消费者使用顺序消费模型，保证同一个队列内的消息串行消费。

---

## 4. 顺序消息的注意事项

### 1. 消费失败会阻塞后续消息

例如同一个订单有三条消息：

```text
订单创建
订单支付
订单发货
```

如果订单支付消息消费失败，订单发货消息不能先消费。

否则就会出现：

```text
还没支付，先发货
```

所以顺序消费失败时，一般不能跳过，只能暂停当前队列，等待重试。

面试表达：

> 顺序消息中，如果前面的消息消费失败，后面的消息不能跳过执行，否则会破坏顺序。因此顺序消息会牺牲一定的并发能力和吞吐量。

---

### 2. 顺序和并发是矛盾的

要保证严格顺序，就不能无限并发。

所以设计时要取舍：

```text
顺序要求越强，并发能力越弱
并发能力越强，越难保证严格顺序
```

面试表达：

> 顺序和并发天然是矛盾的，所以实际项目中通常不会追求全局顺序，而是按照 orderId、userId、accountId 等业务 key 做局部顺序。

---

### 3. 普通重试可能导致乱序

如果使用普通并发消费，可能出现：

```text
消息 1 消费失败
消息 2 消费成功
消息 1 后面重试成功
```

这样业务顺序就乱了。

所以有顺序要求的场景，不能简单使用普通并发消费，需要使用顺序消费模型。

---

# 七、面试标准回答模板

如果面试官问：

## “你们项目中如何保证 MQ 消息可靠性？”

可以回答：

> MQ 消息可靠性我会从生产端、Broker 端、消费端以及业务一致性几个方面来保证。
> 生产端发送消息后不能 fire-and-forget，而是要判断发送结果，失败时进行重试，或者把消息写入本地消息表，后续由定时任务补偿发送。
> 对于业务数据和 MQ 消息一致性，可以使用本地消息表或者 RocketMQ 事务消息。本地消息表是将业务数据和消息记录放在同一个本地事务中提交，后续再异步发送消息，保证消息最终投递。
> Broker 端通过持久化和主从复制保证可靠性。比如 RocketMQ 可以配置同步刷盘和主从同步复制，也就是 SYNC_FLUSH + SYNC_MASTER，可靠性更高，但性能会下降。
> 消费端则要在业务处理成功后再 ACK，如果处理失败就返回失败，让 MQ 重试，多次失败后进入死信队列，再进行人工排查或补偿。

---

## “如果消息被重复消费怎么办？”

可以回答：

> MQ 通常只能保证至少投递一次，不能完全保证只投递一次。为了保证消息不丢，MQ 可能会重复投递，所以消费者必须做好幂等处理。
> 常见幂等方案包括数据库唯一索引、消费记录表、Redis SETNX、业务状态机等。
> 比如订单支付消息，可以通过订单状态机控制，只有待支付状态才能更新为已支付。如果重复收到支付成功消息，由于订单已经是已支付状态，更新影响行数为 0，直接忽略即可。
> 如果是通用消息消费，也可以用 messageId + consumerGroup 建唯一索引，消费前先插入消费记录，插入成功才执行业务，插入失败说明已经消费过，直接返回成功。

---

## “如何保证顺序消息？”

可以回答：

> MQ 一般不保证全局顺序，实际项目中更多是保证局部顺序，比如同一个订单、同一个用户、同一个账户的消息有序。
> 实现方式是生产者发送消息时，根据业务 key 选择队列，比如按照 orderId 取模，让同一个订单的消息始终进入同一个 Queue。消费者端使用顺序消费模型，保证同一个 Queue 内串行消费。
> 但是顺序消息会牺牲并发能力，而且如果前面的消息消费失败，后面的消息不能跳过，否则会导致乱序。因此顺序消息一般只在确实有顺序要求的业务场景中使用。

---

# 八、项目化表达示例

可以结合项目这样说：

> 在项目中，我们使用 MQ 做异步解耦，比如订单创建后发送库存扣减消息、支付成功后发送积分和通知消息。为了保证消息可靠性，生产端不会直接 fire-and-forget，而是会判断 MQ 返回结果，失败时记录日志并进行重试。对于核心业务，会使用本地消息表保证业务数据和消息的一致性。
>
> 消费端处理消息时，会在业务执行成功后再确认消费。如果消费失败，会交给 MQ 重试，超过最大重试次数后进入死信队列。
>
> 为了解决重复消费问题，消费者侧会做幂等控制，比如通过 messageId 建消费记录唯一索引，或者通过订单状态机控制状态流转，避免重复扣库存、重复发券、重复加积分。
>
> 如果业务有顺序要求，比如同一个订单的创建、支付、发货消息必须有序，就会按照 orderId 选择固定队列，保证同一个订单的消息进入同一个 Queue，并使用顺序消费模型串行处理。

---

# 九、核心记忆口诀

```text
可靠性：
发送确认 + 本地消息表/事务消息 + 持久化 + 主从复制 + 手动 ACK + 重试死信

重复消费：
MQ 负责至少投递一次，业务负责幂等

顺序消息：
同一个业务 key 进入同一个队列，单队列串行消费

可靠性最高组合：
同步刷盘 + 主从同步复制

RocketMQ 对应：
SYNC_FLUSH + SYNC_MASTER
```

---

# 十、一句话总结

> MQ 的可靠性不是单点能力，而是生产端、Broker、消费端、业务幂等共同保证的结果。消息可靠性解决的是“不丢”，幂等解决的是“重复不怕”，顺序消息解决的是“同一业务维度不乱序”。
