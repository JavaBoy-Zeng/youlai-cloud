# 企业级 AI 知识库面试题一问一答

## 1. 面试官：如果让你设计一个企业级知识库，你会怎么设计？

**回答：**

如果让我设计一个企业级知识库，我不会只做一个简单的“上传 PDF + 大模型问答”Demo，而是会按照企业级场景来设计。

整体上我会拆成几个核心模块：

```text
1. 知识库管理
2. 文档管理
3. 文档解析
4. 文档切片
5. 向量化索引
6. 混合检索
7. RAG 问答
8. 权限控制
9. 模型网关
10. 审计日志和运维监控
```

整体流程是：

```text
用户上传文档
  ↓
文件存储到 MinIO / NAS
  ↓
文档元数据入库
  ↓
通过 MQ 异步解析
  ↓
文本清洗
  ↓
文档切片 chunk
  ↓
Embedding 向量化
  ↓
写入向量数据库
  ↓
写入 Elasticsearch
  ↓
用户提问时做 BM25 + 向量混合检索
  ↓
Rerank 重排序
  ↓
组装上下文
  ↓
大模型生成答案
  ↓
返回答案和引用来源
```

企业级知识库的重点不是单纯调用大模型，而是要解决**文档入库、检索准确性、权限隔离、文档版本、引用溯源、审计日志和系统可运维**这些问题。

---

## 2. 面试官：企业级知识库和普通知识库 Demo 有什么区别？

**回答：**

普通 Demo 通常只是：

```text
上传文档 -> 向量化 -> 用户提问 -> 向量召回 -> 大模型回答
```

这种方式能演示效果，但离企业级还有差距。

企业级知识库更关注：

```text
1. 多租户隔离
2. 文档权限控制
3. 文档版本管理
4. 文档解析质量
5. 混合检索准确率
6. 引用来源可追溯
7. 问答日志审计
8. 失败重试
9. 模型可切换
10. 数据安全和运维监控
```

所以我理解的企业级知识库，本质上是：

```text
文档管理系统 + 搜索系统 + 权限系统 + RAG 问答系统 + 审计运维系统
```

---

## 3. 面试官：文档上传后，你们是怎么处理的？

**回答：**

文档上传后，我会先把原始文件保存到 MinIO 或 NAS，然后保存文档元数据到数据库，比如文档名称、文件类型、文件大小、知识库 ID、上传人、版本号、解析状态等。

然后通过 MQ 投递异步任务，由消费者执行后续流程：

```text
1. 文档解析
2. 文本清洗
3. 文档切片
4. Embedding 向量化
5. 写入向量数据库
6. 写入 Elasticsearch
7. 更新文档索引状态
```

这样做的原因是文档解析、OCR、切片、向量化都是耗时操作，如果同步处理会导致上传接口阻塞甚至超时。

所以我会用异步任务表记录每个阶段状态，支持失败重试和人工重新索引。

---

## 4. 面试官：为什么文档处理要用 MQ 异步？

**回答：**

因为文档处理链路比较重，尤其是 PDF、Word、Excel、扫描件 OCR、大文件切片和 Embedding 调用，这些都比较耗时。

如果上传接口同步处理，会有几个问题：

```text
1. 接口响应慢
2. 大文件容易超时
3. 用户体验差
4. 失败后不好重试
5. 系统耦合度高
```

所以我会通过 MQ 异步处理。

用户上传文件后，接口只负责保存原始文件和文档元数据，然后投递解析任务。后台消费者负责解析、切片、向量化和索引构建。

同时配合任务状态表：

```text
waiting
running
success
failed
```

这样可以展示处理进度，也可以失败重试。

---

## 5. 面试官：文档解析要注意哪些问题？

**回答：**

文档解析是知识库质量的第一道关。

企业内部文档格式通常比较复杂，比如：

```text
PDF
Word
Excel
PPT
Markdown
TXT
HTML
扫描件图片
接口文档
制度文件
```

解析时不能简单抽纯文本，还要尽量保留结构信息，比如：

```text
标题
段落
表格
页码
Sheet 名称
章节层级
图片 OCR 内容
代码块
```

我会把解析结果保存成 document_block 结构：

```text
document_id
block_type
title_path
content
page_no
sheet_name
sort_no
metadata_json
```

这样后续做切片、引用溯源、上下文组装时更准确。

---

## 6. 面试官：文档切片 chunk 是怎么设计的？

**回答：**

chunk 是知识库检索的最小单位。

切片不能简单按固定字数切，因为太大会引入噪声，太小又会导致语义不完整。

我会结合多种策略：

```text
1. 按标题层级切片
2. 按段落切片
3. 按 token 长度切片
4. 按语义边界切片
5. 表格单独切片
6. FAQ 问答对单独切片
7. 代码块单独切片
```

每个 chunk 会保留元数据：

```text
chunkId
documentId
kbId
tenantId
titlePath
content
pageStart
pageEnd
documentVersion
permissionScope
status
```

其中 titlePath 很重要，比如：

```text
员工制度 / 福利制度 / 年假制度
```

这样即使 chunk 内容较短，也能知道它属于哪个章节，提升召回和回答准确性。

---

## 7. 面试官：什么是向量召回？

**回答：**

向量召回就是把用户问题和文档 chunk 都转成向量，然后根据向量相似度找到语义相关的内容。

流程是：

```text
文档 chunk -> Embedding 模型 -> 向量 -> 存入向量数据库
用户问题 -> Embedding 模型 -> 问题向量 -> 向量数据库相似度检索
```

比如用户问：

```text
离职后年假没休完怎么办？
```

文档中可能没有完全一样的文字，但有：

```text
员工离职时，未休年假应在离职结算中统一处理。
```

通过语义向量检索，可以把这个相关 chunk 召回出来。

---

## 8. 面试官：什么是多文档知识聚合检索？

**回答：**

多文档知识聚合检索指的是：用户提问时，系统不是只查一个文档，而是从多个文档中召回相关片段，然后进行合并、去重、排序和上下文组装。

比如用户问：

```text
员工离职后年假没休完怎么处理？
```

答案可能分散在多个文档里：

```text
员工手册：年假规则
请假制度：未休年假折算
离职流程：离职结算流程
薪酬制度：工资发放时间
```

系统会从这些文档中分别召回相关 chunk，然后聚合成一个上下文，交给大模型综合回答。

所以多文档聚合检索解决的是企业知识分散在多个文件中的问题。

---

## 9. 面试官：什么是 BM25？

**回答：**

BM25 是一种关键词相关性检索算法，常用于 Elasticsearch、Lucene、Solr 这类搜索引擎。

它主要根据几个因素给文档打分：

```text
1. 查询关键词在文档中出现的频率
2. 关键词在整个语料中的稀有程度
3. 文档长度归一化
```

简单来说，就是用户输入一个问题，BM25 会根据关键词匹配程度，从 ES 中找出最相关的文档片段。

比如用户问：

```text
错误码 50013 是什么意思？
```

BM25 对这种错误码、接口名、编号、政策条款等精确匹配场景非常适合。

---

## 10. 面试官：为什么企业知识库不能只用向量检索？

**回答：**

因为向量检索擅长语义相似，但对精确关键词不一定稳定。

比如这些问题：

```text
错误码 50013 是什么意思？
接口 /api/user/list 怎么调用？
合同编号 CQ-2026-001 对应哪个项目？
N+1 补偿怎么计算？
```

这些问题里面有错误码、接口路径、编号、专有词。对于这种精确匹配，BM25 往往比向量检索更稳定。

所以企业知识库一般会采用混合检索：

```text
BM25 关键词检索 + 向量语义检索 + Rerank 重排序
```

BM25 负责精确匹配，向量检索负责语义召回，Rerank 负责重新排序。

---

## 11. 面试官：什么是混合检索？

**回答：**

混合检索就是同时使用关键词检索和向量检索。

流程是：

```text
用户问题
  ↓
Query Rewrite
  ↓
Embedding 向量化
  ↓
Elasticsearch BM25 检索 TopN
  ↓
向量数据库检索 TopN
  ↓
结果合并
  ↓
去重
  ↓
权限过滤
  ↓
Rerank 重排序
  ↓
取 TopK 作为上下文
```

BM25 解决关键词精确匹配问题，向量检索解决语义相似问题。两路召回结果合并之后，再通过 Rerank 排序，选出最适合大模型回答的上下文。

---

## 12. 面试官：如果要用 BM25，切片后是不是也要写入 Elasticsearch？

**回答：**

是的。

如果系统设计了 Elasticsearch BM25 检索，那么文档切片后，每个 chunk 都需要写入 ES。

因为 BM25 是基于倒排索引实现的，ES 需要对 chunk 的 content、titlePath、tags 等字段建立索引，才能根据关键词召回相关片段。

所以切片后通常会有三份数据：

```text
1. 数据库：保存 chunk 主数据
2. Elasticsearch：保存 chunk 文本索引，用于 BM25
3. 向量数据库：保存 chunk 向量，用于语义召回
```

三者通过统一的 chunkId 关联。

---

## 13. 面试官：这样存三份数据，会不会冗余？

**回答：**

会有一定冗余，但这是合理冗余，不是无意义重复。

三份数据的职责不同：

```text
数据库：主数据，保存业务数据、版本、状态、审计
Elasticsearch：关键词索引，用于 BM25 检索
向量数据库：语义索引，用于相似度召回
```

数据库是唯一事实来源，ES 和向量库都是从数据库派生出来的检索索引，是可以重建的。

为了保证一致性，我会采用：

```text
1. 数据库先落库
2. MQ 异步构建 ES 和向量索引
3. 任务状态表记录处理状态
4. 失败重试
5. 文档版本控制
6. active / inactive 状态控制
```

如果是小规模项目，也可以用 PostgreSQL + pgvector 或 Elasticsearch 同时做全文检索和向量检索，减少组件数量。但如果是企业级大规模系统，数据库 + ES + 向量库这种设计更利于性能和扩展。

---

## 14. 面试官：ES 中一条数据应该怎么设计？

**回答：**

ES 中一般是一条 chunk 对应一条文档。

核心字段包括：

```text
chunkId
tenantId
kbId
documentId
documentName
documentVersion
titlePath
content
tags
pageStart
pageEnd
permissionScope
status
createTime
```

其中：

```text
content 用于 BM25 关键词检索
titlePath 可以提高章节相关性
tenantId 用于租户隔离
kbId 用于知识库过滤
permissionScope 用于权限过滤
status 用于过滤无效数据
documentVersion 用于版本控制
pageStart/pageEnd 用于引用溯源
```

这样用户提问时，可以通过 multi_match 查询 titlePath 和 content，同时在 filter 中加上 tenantId、kbId、status、permissionScope 等条件。

## 15. 面试官：BM25 在 Spring Boot 中怎么实现？

**回答：**

可以用两种方式：

```text
1. spring-boot-starter-data-elasticsearch
2. 官方 Elasticsearch Java API Client
```

如果是简单 CRUD，可以用 ElasticsearchRepository。

如果是企业知识库这种复杂检索，我更推荐：

```text
ElasticsearchOperations + NativeQuery
```

或者直接使用官方 Java API Client。

因为知识库检索通常需要控制：

```text
multi_match
字段权重
tenantId 过滤
kbId 过滤
权限过滤
status 过滤
highlight
score
复杂 DSL
```

示例查询逻辑是：

```text
multi_match 查询 titlePath 和 content
titlePath^2 提高标题权重
content^1 正文正常权重
filter 中加入 tenantId、kbId、status、permissionScope
```

ES 默认 text 字段相关性评分就是 BM25，所以不需要自己手写 BM25 算法。

---

## 16. 面试官：为什么不用 Spring Boot Starter，而用官方 Java Client？

**回答：**

不是不能用 Spring Boot Starter，而是要看场景。

如果是简单的 ES 操作，比如保存、删除、普通查询，使用 `spring-boot-starter-data-elasticsearch` 很方便。

但是企业级知识库里的 BM25 检索属于复杂查询场景，需要精细控制：

```text
字段权重
多字段匹配
权限过滤
租户过滤
高亮
score 返回
复杂 DSL
混合检索前的候选召回
```

所以我会优先使用：

```text
ElasticsearchOperations + NativeQuery
```

或者官方 Elasticsearch Java API Client。

面试里可以这样说：

> Repository 适合简单 CRUD，ElasticsearchOperations + NativeQuery 更适合企业项目中的复杂检索，官方 Java Client 则更接近 ES 原生能力，控制力更强。

---

## 17. 面试官：Rerank 是干什么的？

**回答：**

BM25 和向量检索返回的是候选结果，但候选结果排序不一定最准确。

Rerank 的作用是根据用户问题和候选 chunk 内容，重新计算相关性，把真正最符合问题的片段排到前面。

比如 BM25 返回：

```text
chunk_1：年假计算规则
chunk_2：离职结算流程
```

向量检索返回：

```text
chunk_3：薪资发放时间
chunk_2：离职结算流程
```

合并后，Rerank 会重新判断哪个 chunk 对“离职后年假没休完怎么办”这个问题最有帮助，最终可能把“未休年假离职结算规则”排到第一位。

所以 Rerank 可以提高上下文质量，从而提高最终回答准确性。

---

## 18. 面试官：权限控制怎么设计？

**回答：**

企业知识库的权限不能只在前端控制，必须下沉到检索层。

因为如果无权限内容被检索出来并进入 Prompt，大模型就可能把无权限信息回答给用户。

我的设计是：

```text
1. 知识库有权限
2. 文档有权限
3. chunk metadata 中保存权限字段
4. ES 检索时带权限过滤
5. 向量检索时带 metadata filter
6. 合并结果后再做二次权限校验
```

权限字段可以包括：

```text
tenantId
kbId
documentId
deptId
roleId
permissionScope
status
```

这样可以确保用户只能召回自己有权限访问的 chunk。

---

## 19. 面试官：文档更新后，旧向量怎么处理？

**回答：**

文档更新不能简单覆盖，否则容易出现旧知识残留或索引不一致。

我会设计文档版本机制：

```text
document_version = 1
document_version = 2
```

每个 chunk 绑定对应的 documentVersion。

文档更新时：

```text
1. 新版本重新解析
2. 新版本重新切片
3. 新 chunk 写入数据库
4. 新 chunk 写入 ES
5. 新 chunk 写入向量数据库
6. 新版本索引全部成功后，切换为 active
7. 旧版本 chunk 置为 inactive
```

这样可以避免新版本索引失败时知识库不可用，也可以避免用户查到过期内容。

---

## 20. 面试官：如何做引用溯源？

**回答：**

引用溯源的前提是每个 chunk 都要保留来源信息。

比如：

```text
documentId
documentName
documentVersion
titlePath
pageStart
pageEnd
chunkId
blockId
```

大模型回答时，不只是返回 answer，还要返回 references：

```text
引用文档名称
引用页码
引用章节
引用 chunk 内容
```

例如：

```text
根据《员工手册》第12页和《离职流程》第3节，员工离职时未休年假应在离职结算中处理。
```

引用溯源的好处是：

```text
1. 用户可以验证答案
2. 方便审计
3. 降低幻觉风险
4. 出错时可以定位问题
5. 方便优化知识库
```

---

## 21. 面试官：如何防止大模型幻觉？

**回答：**

我会从检索和生成两个阶段控制。

检索阶段：

```text
1. 提高文档解析质量
2. 优化切片
3. 使用 BM25 + 向量混合检索
4. Rerank 重排序
5. 过滤低相关内容
6. 过滤无权限内容
```

生成阶段：

```text
1. Prompt 限制模型只能基于上下文回答
2. 没有依据时必须拒答
3. 重要结论必须带引用来源
4. 禁止编造制度、金额、时间、接口参数
5. 如果文档冲突，要提示存在冲突
```

比如 Prompt 可以写：

```text
你是企业知识库助手。
你只能根据提供的上下文回答。
如果上下文没有答案，请回答“知识库中未找到相关依据”。
不要编造制度、金额、时间、法律条款和接口参数。
回答必须包含引用来源。
```

企业级知识库宁可回答“知识库中未找到依据”，也不能胡编。

---

## 22. 面试官：如何提高知识库回答准确率？

**回答：**

我会从五个方面提高准确率：

第一，提升文档入库质量。
包括文档解析、OCR、文本清洗、表格结构化、页眉页脚清理等。

第二，优化切片策略。
chunk 不能太大也不能太小，要结合标题、段落、语义边界、表格结构来切，并保留 titlePath。

第三，使用混合检索。
通过 BM25 解决关键词精确匹配，通过向量检索解决语义相似，再通过 Rerank 重排序提升最终上下文质量。

第四，控制上下文质量。
过滤低相关、重复、过期、无权限的 chunk，只把高质量 TopK 上下文交给大模型。

第五，限制模型生成。
Prompt 要限制模型只能基于上下文回答，没有依据就拒答，并且返回引用来源。

最后还要做用户反馈闭环，根据用户反馈持续优化文档、切片、召回和 Prompt。

---

## 23. 面试官：Query Rewrite 有什么作用？

**回答：**

Query Rewrite 是问题改写，主要用于提升召回率。

用户的问题经常很口语化，比如：

```text
离职后年假咋算？
```

系统可以改写成：

```text
员工离职时未休年假如何结算？
离职未休年假折算规则是什么？
离职结算中年假如何处理？
```

这样可以提高 BM25 和向量检索的召回效果。

不过 Query Rewrite 只能扩展用户问题，不能改变用户原意，否则会导致错误召回。

## 24. 面试官：表格类文档怎么处理？

**回答：**

表格不能简单当成普通文本处理，因为会丢失行列关系。

比如：

```text
部门 | 预算 | 实际支出
研发部 | 100万 | 80万
```

如果直接抽成一段文本，大模型可能理解不准确。

我会对表格单独解析，保留：

```text
表头
行数据
列数据
Sheet 名称
页码
合并单元格信息
```

然后可以把每一行转换成自然语言：

```text
研发部的预算是100万，实际支出是80万。
```

对于复杂统计类问题，可以把表格结构化入库，再通过 SQL 查询或 SQL Agent 处理。

kb_knowledge_base：知识库表

kb_document_version：文档版本表
kb_document_block：文档解析块表
kb_chunk：文档切片表

kb_index_task：索引任务表
kb_chat_session：会话表

其中 `kb_chunk` 是检索核心表，保存切片文本和元数据。

## 26. 面试官：模型网关为什么要单独设计？

**回答：**

企业级系统不能和某一个大模型厂商强绑定。

所以我会设计模型网关，统一封装 Chat 模型和 Embedding 模型。

接口可以抽象为：

```java
public interface LlmClient {
    ChatResult chat(ChatRequest request);

    List<Float> embed(String text);
}
```

这样业务层不关心底层调用的是 OpenAI、通义千问、智谱、DeepSeek 还是本地私有化模型。

模型网关可以实现：

```text
1. 模型切换
2. 流式输出
3. 超时控制
4. 失败重试
5. 限流熔断
6. 成本统计
7. 多模型路由
8. 私有化模型接入
```

## 27. 面试官：如何处理检索结果的数据一致性？

**回答：**

数据库是主数据，ES 和向量库是派生索引。

我会采用：

```text
1. 数据库先落库
2. 保存 chunk 主数据
3. 发送 MQ 索引任务
4. 异步写入 ES
5. 异步写入向量数据库
6. 更新 index_status
7. 失败后重试
```

如果 ES 或向量库写入失败，不影响主数据，但该 chunk 暂时不参与检索。

同时通过任务表记录失败原因，支持自动重试和人工重新索引。

## 28. 面试官：知识库里哪些内容适合 BM25，哪些适合向量检索？

**回答：**

BM25 适合精确匹配，比如：

```text
错误码
接口路径
合同编号
政策条款
专有名词
金额
日期
文件标题
```

向量检索适合语义匹配，比如：

```text
用户口语化表达
同义词
模糊问题
跨文档语义关联
文档说法和用户说法不一致
```

例如：

```text
错误码 50013 是什么意思？
```

适合 BM25。

而：

```text
公司把我辞退了要赔多少钱？
```

可能文档里写的是：

```text
解除劳动合同经济补偿标准
```

这种更适合向量检索。

所以实际项目里我会结合两种检索方式。

## 29. 面试官：TopN 和 TopK 有什么区别？

**回答：**

TopN 通常是召回阶段的候选数量。

比如：

```text
BM25 检索 TopN = 20
向量检索 TopN = 20
```

两路召回之后可能有 40 条候选结果，经过合并、去重、权限过滤和 Rerank 后，再取最终 TopK。

TopK 是最终进入大模型上下文的数量，比如：

```text
最终 TopK = 5
```

也就是说：

```text
TopN 是先多召回一些候选
TopK 是最终筛选后给大模型使用的高质量上下文
```

## 30. 面试官：如何判断没有找到答案？

**回答：**

可以通过多个信号判断：

```text
1. BM25 分数低
2. 向量相似度低
3. Rerank 分数低
4. TopK 数量不足
5. 召回内容与问题主题不匹配
```

如果低于阈值，就不应该让大模型硬答，而是返回：

```text
当前知识库中未检索到足够相关的依据，建议补充相关文档或换一种问法。
```

这样可以减少大模型幻觉。

## 31. 面试官：企业级知识库的难点在哪里？

**回答：**

我觉得企业级知识库的难点不在于调用大模型接口，而在于如何让大模型基于企业内部知识，回答得准确、可控、安全、可追溯。

主要难点有：

```text
1. 文档质量不稳定
2. 文档解析和清洗复杂
3. 切片策略影响召回
4. 单纯向量检索不够准
5. 多文档知识聚合复杂
6. 权限控制必须下沉到检索层
7. 文档更新后要避免旧知识残留
8. 大模型幻觉需要控制
9. 答案必须引用溯源
10. 上线后需要反馈闭环持续优化
```

所以企业知识库不是一次性开发完，而是一个持续运营和优化的 RAG 工程体系。

# 最终面试背诵版

如果面试官让你整体介绍，可以这样回答：

> 我们设计企业级知识库时，不是简单做一个文档问答 Demo，而是按照企业真实场景来设计。整体包括文档管理、文档解析、切片向量化、混合检索、RAG 问答、权限控制、模型网关、审计日志和运维监控几个模块。
>
> 文档上传后，先保存到 MinIO 或 NAS，同时将文档元数据入库，然后通过 MQ 异步任务进行文档解析、文本清洗、chunk 切片、Embedding 向量化和索引构建。切片后的 chunk 会同时写入数据库、Elasticsearch 和向量数据库，其中数据库保存主数据，ES 用于 BM25 关键词检索，向量数据库用于语义召回，三者通过 chunkId 关联。
>
> 用户提问时，系统会先做 Query Rewrite，然后一方面通过 ES 做 BM25 关键词召回，另一方面通过向量数据库做语义召回。两路结果合并后，会进行去重、权限过滤和 Rerank 重排序，最终取 TopK 片段组装成上下文交给大模型生成答案，并返回引用来源。
>
> 企业级场景里我重点关注几个问题：第一是文档解析和切片质量；第二是检索准确率，所以要做 BM25 + 向量检索 + Rerank；第三是权限安全，权限必须下沉到检索层，不能让无权限内容进入 Prompt；第四是文档版本管理，避免旧知识残留；第五是引用溯源和审计日志，保证答案可验证、可追踪。
>
> 所以我认为企业级知识库的核心不是调用模型，而是围绕文档入库、混合检索、权限控制、上下文组织、引用溯源和反馈优化构建一套完整的 RAG 工程体系。
